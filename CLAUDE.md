# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Marketing website for Ready Signal (readysignal.com) — a forecasting platform that provides external data signals. React 18 SPA with TypeScript, Vite, and Tailwind CSS. Features AI chatbot (Supabase Edge Functions + OpenAI RAG pipeline), hybrid blog system (Supabase + static fallback), and markdown-driven forecasting content. Not the product app — this is the public marketing site.

## Commands

```bash
npm install              # Install dependencies
npm run dev              # Vite dev server (http://localhost:5173)
npm run build            # Production build (pre: sitemap, post: OG images)
npm run preview          # Preview production build locally
npm run lint             # ESLint for TS/TSX
npm run typecheck        # TypeScript type checking (tsc --noEmit)
npm run granger          # Python Granger causality analysis script
```

Deployed on **Netlify**. Build command: `npm run build`. Publish dir: `dist`.

## Architecture

**Routing**: React Router v7 in `src/App.tsx` with 260+ routes. Catch-all `/:slug` at the bottom handles blog articles — new routes must go ABOVE it or they'll be swallowed.

**Hybrid Article System**: Articles load from Supabase first; fall back to static data in `src/data/blogArticles.ts` when Supabase is unavailable. Static data is auto-generated by `scripts/migrate-blogs.mjs` — never edit it by hand.

**Forecasting Content**: Markdown files imported via Vite's `?raw` loader, rendered with `react-markdown` + remark-gfm. No admin UI needed.

**Chatbot (RAG)**: `ChatContext` manages state. Posts to `{SUPABASE_URL}/functions/v1/chat`. Knowledge base uses pgvector embeddings (OpenAI text-embedding-3-small), LLM is GPT-4-turbo.

**Supabase Backend**: Edge functions in `supabase/functions/` (chat, ingest-document, ai-enhance, generate-article, research, admin). Migrations in `supabase/migrations/`.

**Build Scripts** (`scripts/`): `generate-sitemap.mjs` (pre-build), `generate-og-pages.mjs` (post-build), blog migration utilities.

**Analytics**: GA4 via `react-ga4`. Route changes tracked by `<RouteChangeTracker>`. Helpers in `src/lib/analytics.ts`.

## Key Conventions

- **Styling**: Tailwind CSS exclusively. No inline styles, no CSS modules.
- **Icons**: Lucide React exclusively. Do not install other icon packages.
- **Animation**: Framer Motion (`motion.div`, `AnimatePresence`).
- **Components**: Functional components, default exports, PascalCase filenames matching component name.
- **SEO**: Every page uses the custom `<SEO>` component for meta tags. No page ships without it.
- **Theme colors**: `rs-dark: #404C57`, `rs-yellow: #FCB900`, `rs-cyan: #0693E3`, `rs-light-gray: #F9FAFB` (in `tailwind.config.js`).
- **Error handling**: Soft fallbacks everywhere. No Supabase → use static data. Missing env vars → console warn, don't crash.

## Environment Variables

**Required**: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`

**Optional**: `VITE_GA_MEASUREMENT_ID` (site works without it)

All client-side env vars must be prefixed `VITE_` or Vite won't expose them.

**Supabase Edge Function secrets** (set via `supabase secrets set`): `OPENAI_API_KEY`, `PERPLEXITY_API_KEY`

If Supabase vars are missing, the client creates a placeholder (`placeholder.supabase.co`). Chat won't work but the site renders fine with static fallback data.

## Known Gotchas

- **`blogArticles.ts` is auto-generated** — never edit manually. Regenerate via `node scripts/migrate-blogs.mjs`.
- **Netlify `_redirects`** has 50+ SEO rules from the WordPress migration. Check before adding routes that might conflict with legacy URL patterns.
- **Catch-all route** (`/:slug`) at the bottom of App.tsx — new routes added below it will never match.
- **Placeholder Supabase client** — missing env vars create a dummy client. Site renders but Supabase calls silently fail. Intentional.
- **Post-build OG generation** — if `generate-og-pages.mjs` fails, the build still succeeds but social sharing images won't generate.

## Code Quality Rules

**Never delete code, remove dependencies, stub functions, or reduce scope to resolve errors.** Fix the actual problem. Diagnose the root cause — don't work around it by removing functionality.

**Never add features, refactoring, or "improvements" beyond what was asked.** A bug fix doesn't need surrounding code cleaned up. Don't add docstrings, comments, or type annotations to code you didn't change.

**Don't add error handling for scenarios that can't happen.** Trust internal code and framework guarantees. Only validate at system boundaries (user input, external APIs, webhook payloads).

**Don't create abstractions for one-time operations.** Three similar lines of code is better than a premature helper function.

**Match the codebase.** Follow existing patterns for naming, error handling, file organization, and framework usage. Don't introduce new libraries or patterns without being asked.

## Verification & QA

**Self-reporting is not proof. Filesystem evidence is.**

Before declaring any task complete, you must show at least one of:
- **Terminal output** — actual stdout/stderr from running the code
- **Filesystem confirmation** — `ls -la`, `cat`, or `find` proving files exist
- **Screenshot** — of the running UI or browser state (required for any UI change)

**You are not allowed to:**
- Say "the file was created" without showing it exists
- Say "the UI looks correct" without a screenshot
- Say "the test passed" without showing test runner output
- Close a task based on code review alone — reading code is not executing code

**You are not done until:**
1. The code executed without errors
2. The output is actually correct — not just "no errors"
3. No stubs or placeholder code exists in the feature you touched
4. You can show terminal or filesystem proof of the above

**Testing means executing, not reading.** NEVER report "looks good" without having actually run the code. If you can't run it, say so explicitly. When you find a problem — fix it, show the diff, re-run to confirm.

## Build & Deploy Safety

After every push to a deploy branch:
1. Check CI status — don't move on until the build is green
2. If the build fails, fix it before doing anything else
3. Confirm the Netlify deploy succeeds (check deploy logs)
4. A task is NOT done until the build is green and the deployment is verified healthy

**Rollback:** prefer `git revert` over `git reset --hard`. Never force-push to main/master.

## Working With Claude Code — Tips

- Run `/compact` manually at ~50% context usage — don't wait for auto-compaction.
- Break large tasks into subtasks that can each complete in under 50% context.
- Start complex tasks in plan mode. Pour energy into the plan so Claude can one-shot the implementation.
- When execution goes sideways, stop and re-plan. Don't keep pushing a failing approach.
- Give Claude a way to verify its work (run the build, screenshot the UI). This 2-3x the quality.
- After a mediocre fix, try: "Knowing everything you know now, scrap this and implement the elegant solution."
